# 需求分析

---

## 需求一：文本转语音生成

### 需求描述
用户输入文本内容，选择语音音色，生成对应的语音音频文件。系统支持多种生成参数配置（temperature、topP、topK、repetitionPenalty），生成完成后用户可以在线播放、下载或查看生成历史。

### 实现思路

**前端：**
1. 使用TanStack Form管理表单状态，Zod进行输入验证
2. 提供文本输入框、语音选择器、参数滑块组件
3. 提交表单调用tRPC的generations.create mutation
4. 生成完成后跳转到详情页面展示音频播放器

**后端：**
1. tRPC Router接收文本、voiceId及生成参数
2. 验证用户订阅状态（调用Polar API）
3. 从数据库获取语音的R2存储路径
4. 调用Modal部署的Chatterbox TTS服务生成音频
5. 将生成的音频上传到R2存储
6. 在数据库创建Generation记录并上报用量到Polar

### 处理流程

```
用户输入文本 → 前端表单验证 → tRPC调用 → 验证订阅状态 
→ 获取语音文件路径 → 调用TTS服务 → 上传音频到R2 
→ 保存生成记录 → 返回Generation ID → 前端跳转详情页
```

## 需求二：语音克隆

### 需求描述
用户可以上传自己的音频样本或实时录音，系统基于该音频克隆出个性化的语音音色。克隆的音色会自动添加到用户的语音库中，可用于后续的TTS生成。克隆需要至少10秒的音频样本。

### 实现思路

**前端：**
1. 支持两种音频输入方式：文件上传和浏览器录音
2. 录音功能使用RecordRTC + WaveSurfer.js实现实时波形可视化
3. 上传前使用music-metadata解析音频文件，验证格式和时长
4. 调用API路由 POST /api/voices/create 提交克隆请求

**后端：**
1. 验证用户订阅状态
2. 校验音频文件大小（最大20MB）和时长（最少10秒）
3. 生成Voice记录，获取orgId
4. 将音频文件上传到R2存储：voices/orgs/{orgId}/{voiceId}
5. 更新Voice记录的r2ObjectKey
6. 上报语音创建用量到Polar

### 处理流程

```
用户上传/录制音频 → 前端校验文件格式和时长 → 调用API 
→ 验证订阅 → 校验音频 → 上传到R2 → 创建Voice记录 
→ 更新r2ObjectKey → 上报用量 → 返回成功
```

## 需求三：用户认证与多租户

### 需求描述
系统基于Clerk实现用户认证，用户注册后可以创建或加入组织（Organization）。每个组织拥有独立的语音库和生成历史，组织间数据完全隔离。

### 实现思路

**认证：**
1. 使用@clerk/nextjs提供登录/注册页面
2. 中间件保护dashboard路由，未登录用户重定向到登录页

**多租户：**
1. Clerk Organizations支持团队协作
2. tRPC的orgProcedure中间件从请求上下文提取orgId
3. 所有数据库查询自动注入orgId过滤条件
4. 语音查询区分SYSTEM（全局）和CUSTOM（组织私有）

### 处理流程

```
用户访问 → 检查Clerk session → 无登录则跳转登录页 
→ 登录成功后检查Organization → 无则引导创建/选择 
→ 后续请求通过中间件注入orgId → 路由级别数据隔离
```

## 需求四：订阅与支付

### 需求描述
系统采用按量计费模式，用户需要订阅后才能使用TTS生成和语音克隆功能。订阅基于Polar实现，支持Metered Pricing按实际使用量计费。

### 实现思路

**订阅流程：**
1. 用户点击订阅按钮，调用billing.createCheckout创建结账会话
2. 跳转到Polar支付页面完成支付
3. 支付成功后自动激活订阅

**用量计费：**
1. Polar配置两种Meter：TTS字符数和语音创建次数
2. 每次TTS生成后上报字符数到Polar
3. 每次语音创建后上报次数到Polar
4. 用户可在账单页面查看预估费用

**管理功能：**
1. 用户可通过billing.createPortalSession进入客户门户
2. 查看订阅状态、账单历史、管理付款方式

### 处理流程

```
用户点击订阅 → 调用Polar创建Checkout → 跳转到支付页 
→ 支付成功 → Polar webhook激活订阅 → TTS生成时验证订阅 
→ 上报用量到Polar → 用户在账单页查看费用
```

## 需求五：音频播放与波形可视化

### 需求描述
生成的音频需要支持在线播放，用户可以播放、暂停、拖拽进度、跳转前后5秒。播放界面需要展示音频波形，支持可视化交互。

### 实现思路

**音频播放器：**
1. 使用WaveSurfer.js渲染波形
2. 支持进度拖拽、播放/暂停、前后跳转
3. 实时显示当前播放时间和总时长
4. 响应式设计：移动端使用独立全屏播放器

**音频访问：**
1. R2存储的音频通过API路由代理：`/api/audio/[generationId]`
2. 使用预签名URL（1小时有效期）实现安全访问
3. API路由根据generationId查询R2 ObjectKey，返回流式音频

### 处理流程

```
用户进入详情页 → 加载generation数据 → 通过API路由获取音频 
→ 初始化WaveSurfer → 渲染波形 → 用户点击播放 
→ 音频流式播放 → 实时更新播放进度 → 支持拖拽跳转
```

## 需求六：生成历史管理

### 需求描述
用户可以查看所有历史生成记录，支持按语音名称搜索。历史记录保留语音参数配置，用户可以重新生成相同内容的音频。

### 实现思路

**查询：**
1. tRPC的generations.getAll端点返回当前组织的所有记录
2. 按创建时间倒序排列
3. 不返回r2ObjectKey（由前端按需通过API获取）

**展示：**
1. 列表页展示每条记录的文本摘要、语音名称、创建时间
2. 点击跳转到详情页，可重新播放或复制参数生成新音频
3. 详情页保留原始生成参数，可作为新生成的默认值

### 处理流程

```
用户进入历史页 → 调用generations.getAll → 返回列表 
→ 用户点击某条记录 → 跳转到详情页 → 加载完整信息 
→ 播放音频或修改参数重新生成
```

## 需求七：语音库管理

### 需求描述
用户可以管理自己创建的语音音色，包括查看、搜索、删除操作。系统预置20种官方音色对所有用户可见，用户自定义音色仅自己可见。

### 实现思路

**数据结构：**
1. Voice表通过variant字段区分SYSTEM和CUSTOM
2. SYSTEM音色orgId为空，CUSTOM音色orgId为所属组织

**查询：**
1. voices.getAll并行查询SYSTEM和CUSTOM音色
2. 支持按名称和描述搜索（不区分大小写）
3. SYSTEM按名称排序，CUSTOM按创建时间倒序

**删除：**
1. 验证音色属于当前组织且variant为CUSTOM
2. 删除数据库记录
3. 异步删除R2存储的音频文件

### 处理流程

```
用户进入语音库 → 并行查询SYSTEM和CUSTOM音色 → 返回合并结果 
→ 用户搜索 → 模糊匹配名称/描述 → 用户删除音色 
→ 验证归属 → 删除DB记录 → 异步删除R2文件
```

# Resonance 项目逆向工程分析报告

## 1. 核心用户需求总结

根据代码分析，Resonance 项目满足以下核心用户需求：

### 1.1 语音克隆（Voice Cloning）
- **需求**：用户可以通过上传或录制 10 秒以上的音频片段，创建自定义语音
- **场景**：品牌声音、个人声音、角色声音的创建和复用
- **实现**：支持文件上传和麦克风录制两种方式，音频文件存储在 R2，语音信息存入数据库

### 1.2 文本转语音（Text-to-Speech）
- **需求**：输入文本，选择语音（系统预设或自定义），生成高质量音频
- **场景**：有声读物、播客、广告配音、内容创作
- **实现**：调用 Chatterbox AI API 进行语音合成，支持温度、topP、topK 等参数调整

### 1.3 多租户协作
- **需求**：支持团队协作，语音和生成记录按组织隔离
- **场景**：企业团队共享语音库、多用户协作创作
- **实现**：基于 Clerk 的组织管理，数据库中每个记录关联 orgId

### 1.4 订阅计费
- **需求**：按使用量计费（字符数），支持订阅管理和用量统计
- **场景**：SaaS 商业模式，付费用户升级
- **实现**：集成 Polar 平台，订阅检查、用量跟踪、计费事件上报

### 1.5 语音管理
- **需求**：浏览、搜索、试听、删除自定义语音
- **场景**：语音库的维护和管理
- **实现**：语音列表展示，支持按名称/描述搜索，实时试听

### 1.6 生成历史
- **需求**：查看所有生成记录，支持重新生成相同配置
- **场景**：找回历史作品、批量生成
- **实现**：Generation 模型记录所有生成，保留参数快照

### 1.7 音频预览
- **需求**：实时播放语音，可视化波形
- **场景**：生成前试听、生成后预览
- **实现**：Wavesurfer.js 可视化，支持播放/暂停控制

## 3. 架构设计亮点

### 3.1 多租户数据隔离
- 每个 Voice 和 Generation 记录都关联 orgId
- API 层统一进行组织权限检查（orgProcedure）
- 数据库层面通过索引优化查询性能

### 3.2 类型安全全栈
- tRPC 实现前后端共享类型
- Prisma 生成 TypeScript 模型
- Zod 用于运行时验证
- openapi-fetch 为第三方 API 提供类型

### 3.3 服务端渲染优化
- 服务端预取数据（prefetch）
- React 19 Suspense 边界
- HydrationBoundary 水合优化
- Cache 控制音频响应

### 3.4 计费与订阅
- 使用前检查订阅状态
- Polar Meter 跟踪使用量
- 事件驱动的用量上报
- 客户门户管理订阅

## 4. 关键业务流程

### 4.1 创建自定义语音流程
```
1. 用户选择上传或录制音频
2. 前端验证：文件类型、大小（20MB）、时长（≥10s）
3. 提交到 /api/voices/create
4. 后端验证订阅状态
5. 保存音频到 R2
6. 创建 Voice 记录
7. 上报用量事件到 Polar
8. 刷新语音列表
```

### 4.2 文本转语音流程
```
1. 用户输入文本，选择语音
2. 提交表单触发 generations.create
3. 后端验证订阅状态
4. 获取语音音频路径（r2ObjectKey）
5. 调用 Chatterbox API 生成音频
6. 保存音频到 R2
7. 创建 Generation 记录
8. 上报用量事件（字符数）
9. 返回生成记录 ID
10. 跳转到详情页面
```

### 4.3 音频播放流程
```
1. 用户点击播放按钮
2. 请求 /api/audio/[generationId] 或 /api/voices/[voiceId]
3. 后端验证权限（用户/组织）
4. 从 R2 获取签名 URL
5. 代理音频流（设置 Cache-Control）
6. 前端使用 Wavesurfer 播放和显示波形
```

## 5. 数据模型关系

### Voice（语音）
```typescript
{
  id: string,           // 语音唯一标识
  orgId: string?,       // 所属组织（SYSTEM 为 null）
  name: string,         // 语音名称
  description: string?, // 描述
  category: VoiceCategory, // 分类（12 种）
  language: string,     // 语言（如 en-US）
  variant: "SYSTEM" | "CUSTOM", // 系统/自定义
  r2ObjectKey: string?, // 音频文件路径
  generations: Generation[] // 关联的生成记录
}
```

### Generation（生成记录）
```typescript
{
  id: string,           // 生成记录唯一标识
  orgId: string,        // 所属组织
  voiceId: string?,     // 使用的语音 ID
  text: string,         // 生成文本
  voiceName: string,    // 语音名称快照（防止被删除后丢失）
  r2ObjectKey: string?, // 音频文件路径
  temperature: float,   // 生成参数
  topP: float,
  topK: int,
  repetitionPenalty: float,
  createdAt: DateTime
}
```

## 6. 安全与权限控制

### 6.1 认证层级
1. **baseProcedure**：无认证，仅用于公开数据（项目中未使用）
2. **authProcedure**：需要登录用户（userId）
3. **orgProcedure**：需要登录用户且选择组织（userId + orgId）

### 6.2 数据访问控制
- 自定义语音：仅所属组织成员可访问
- 系统语音：所有用户可访问
- 生成记录：仅创建组织成员可访问
- 删除操作：仅语音创建者（组织）可删除

### 6.3 订阅检查
- 创建语音前检查订阅
- 生成音频前检查订阅
- 无订阅时提示升级并跳转到 Polar 结账

## 7. 性能优化策略

### 7.1 缓存策略
- 系统语音音频：public, max-age=86400（1 天）
- 自定义语音/生成音频：private, max-age=3600（1 小时）
- React Query 数据缓存：按需配置

### 7.2 预取与水合
- 服务端预取数据（prefetch）
- SSR 水合避免客户端重复请求
- Suspense 边界优化加载状态

### 7.3 批量请求
- tRPC HTTP Batch 链接
- 减少网络请求次数

## 8. 扩展性设计

### 8.1 可扩展的语音分类
- 12 种预定义分类（可扩展）
- 枚举类型确保数据一致性
- 前端动态映射标签

### 8.2 用量计量
- Polar Meter 支持多种计量维度
- 当前使用：字符数（tts_generation）、语音创建（voice_creation）
- 易于添加新的计量项

### 8.3 多语言支持
- 使用 locale-codes 库
- 语音模型支持 language 字段
- 前端国际化准备（regionNames）

## 9. 限制与约束

### 9.1 业务限制
- 文本长度：最大 5000 字符
- 音频文件：最大 20MB
- 自定义语音：最短 10 秒
- 订阅要求：创建语音和生成音频都需要订阅

### 9.2 技术限制
- 仅支持 wav 格式音频输出
- 依赖 Chatterbox API 进行语音合成
- 依赖 Polar 进行订阅管理

## 2.2 服务端初始化详解

让我们先看看项目的核心文件 `src/trpc/init.ts`：

```typescript
import * as Sentry from "@sentry/node";
import { auth } from '@clerk/nextjs/server';
import { initTRPC, TRPCError } from '@trpc/server';
import { cache } from 'react';
import superjson from "superjson";
```

**引入的库说明：**
- `@trpc/server` - tRPC 核心库
- `@sentry/node` - 错误监控和追踪
- `@clerk/nextjs/server` - 用户认证
- `superjson` - 数据序列化
- `react` - 使用 cache 函数

### 2.2.1 创建 Context

```typescript
export const createTRPCContext = cache(async () => {
  /**
   * @see: https://trpc.io/docs/server/context
   */
  return {};
});
```

**为什么要用 cache？**

`cache` 是 React 提供的一个函数，它确保在同一个请求周期内，函数只会被调用一次。

```typescript
// 不使用 cache 的问题
function createTRPCContext() {
  console.log('Context created');  // 每次调用都会打印
  return {};
}

// 使用 cache 后
export const createTRPCContext = cache(async () => {
  console.log('Context created');  // 同一个请求只打印一次
  return {};
});
```

**在项目中的实际使用：**

在这个项目中，Context 是空的，认证信息是在中间件中处理的。这种设计让 Context 保持简洁，认证逻辑集中在中间件中。

### 2.2.2 初始化 tRPC

```typescript
const t = initTRPC.create({
  /**
   * @see https://trpc.io/docs/server/data-transformers
   */
  transformer: superjson,
});
```

**superjson 的作用：**

superjson 让我们可以自动处理 JavaScript 中特殊的数据类型：

```typescript
// 普通 JSON 的问题
const data = {
  createdAt: new Date(),  // Date 对象
  bigNumber: BigInt(123456789),  // BigInt
  undefined: undefined,  // undefined 会被忽略
};

// 使用 superjson 后，这些类型都能正确序列化和反序列化
const serialized = superjson.serialize(data);
// { createdAt: "2024-01-01T00:00:00.000Z", bigNumber: "123456789" }

const deserialized = superjson.deserialize(serialized);
// { createdAt: Date, bigNumber: BigInt }
```

**为什么需要 superjson？**

1. **Date 对象** - JSON 不能直接传输 Date，需要转成字符串
2. **BigInt** - JSON 不支持 BigInt
3. **Set/Map** - JSON 不支持
4. **类实例** - 需要自定义序列化

### 2.2.3 导出基础工具

```typescript
// Base router and procedure helpers
export const createTRPCRouter = t.router;
export const createCallerFactory = t.createCallerFactory;
export const baseProcedure = t.procedure.use(sentryMiddleware);
```

这三个导出是构建 tRPC 应用的基础：

| 导出 | 作用 |
|------|------|
| `createTRPCRouter` | 创建 Router 的工厂函数 |
| `createCallerFactory` | 用于服务内部调用 |
| `baseProcedure` | 基础 Procedure，添加了 Sentry 中间件 |

## 2.3 中间件设计

中间件是 tRPC 的强大功能，让我们可以为 Procedure 添加前置处理逻辑。

### 2.3.1 Sentry 中间件

```typescript
const sentryMiddleware = t.middleware(
  Sentry.trpcMiddleware({
    attachRpcInput: true,
  }),
);
```

**Sentry 是什么？**

Sentry 是一个错误监控平台，可以帮助我们：
- 追踪应用中的错误
- 查看错误发生的上下文
- 分析错误发生的频率

**attachRpcInput 的作用：**

当发生错误时，将请求的输入参数附加到 Sentry 事件中：

```typescript
// 没有 attachRpcInput
// 错误事件只有：Error: Not found

// 有 attachRpcInput
// 错误事件包含：Error: Not found
// 上下文：{ input: { id: "123" }, path: "voices.delete" }
```

这样调试生产环境错误时，我们可以知道用户传入的具体参数。

### 2.3.2 认证中间件 (authProcedure)

```typescript
export const authProcedure = baseProcedure.use(async ({ next }) => {
  const { userId } = await auth();

  if (!userId) {
    throw new TRPCError({ code: "UNAUTHORIZED" });
  }

  return next({
    ctx: { userId },
  });
});

// 显示写法

const isAuth = t.middleware(async ({ next }) => {
  const { userId } = await auth()

  if(!userId) {
    throw new TRPCError({ code: "UNAUTHORIZED" })
  }

  return next(
    ctx:{ userId },
  )
})

export const authProcedure = baseProcedure.use(isAuth)
```

**执行流程：**

```
1. 请求进入 authProcedure
     ↓
2. 调用 await auth() 获取用户信息
     ↓
3. 检查 userId 是否存在
     ↓
4. 不存在 → 抛出 UNAUTHORIZED 错误
   存在 → 继续执行
     ↓
5. 将 userId 注入 ctx
     ↓
6. 执行后续业务逻辑
```

**auth() 来自哪里？**

`auth` 来自 `@clerk/nextjs/server`，Clerk 是一个用户认证平台。

```typescript
import { auth } from '@clerk/nextjs/server';

const { userId, orgId, sessionId } = await auth();

// userId: 当前用户的 ID
// orgId: 当前组织 ID（如果是组织成员）
// sessionId: 当前会话 ID
```

### 2.3.3 组织权限中间件 (orgProcedure)

```typescript
export const orgProcedure = baseProcedure.use(async ({ next }) => {
  const { userId, orgId } = await auth();

  if (!userId) {
    throw new TRPCError({ code: "UNAUTHORIZED" });
  }

  if (!orgId) {
    throw new TRPCError({
      code: "FORBIDDEN",
      message: "Organization required",
    });
  }

  return next({ ctx: { userId, orgId } });
});
```

**为什么需要 orgProcedure？**

在这个项目中，用户的操作是按组织（Organization）隔离的：

- 每个组织有自己的声音（Voices）
- 每个组织有自己的语音生成记录（Generations）
- 只有组织成员才能访问组织资源

**执行流程：**

```
1. 请求进入 orgProcedure
     ↓
2. 调用 await auth() 获取用户和组织信息
     ↓
3. 检查 userId（必须登录）
     ↓
4. 检查 orgId（必须是组织成员）
     ↓
5. 验证通过 → 将 userId 和 orgId 注入 ctx
     ↓
6. 执行后续业务逻辑
```

### 2.3.4 中间件链式调用

```
请求 → baseProcedure → authProcedure → orgProcedure → 业务逻辑
                  ↓              ↓             ↓
              Sentry追踪     用户认证       组织权限
```

**实际代码示例：**

当我们使用 `orgProcedure` 定义 API 时：

```typescript
// 这个 API 会经过所有中间件
export const voicesRouter = createTRPCRouter({
  getAll: orgProcedure
    .query(async ({ ctx }) => {
      // ctx 包含 { userId, orgId }
      // 已经通过了 Sentry追踪 + 用户认证 + 组织权限
      
      const voices = await prisma.voice.findMany({
        where: { orgId: ctx.orgId }  // 只查询当前组织的
      });
      
      return voices;
    }),
});
```

## 2.4 错误处理

tRPC 使用 `TRPCError` 来处理错误：

```typescript
import { TRPCError } from '@trpc/server';

throw new TRPCError({
  code: "UNAUTHORIZED",    // 错误码
  message: "User not found", // 错误消息
});
```

**常用错误码：**

| 错误码 | 含义 | HTTP 对应 |
|--------|------|-----------|
| `UNAUTHORIZED` | 未登录 | 401 |
| `FORBIDDEN` | 无权限 | 403 |
| `NOT_FOUND` | 资源不存在 | 404 |
| `BAD_REQUEST` | 请求参数错误 | 400 |
| `INTERNAL_SERVER_ERROR` | 服务器内部错误 | 500 |
| `TOO_MANY_REQUESTS` | 请求过多 | 429 |

## 2.5 API 端点处理

最后，请求是如何被处理的呢？让我们看看 `src/app/api/trpc/[trpc]/route.ts`：

```typescript
import { fetchRequestHandler } from '@trpc/server/adapters/fetch';
import { createTRPCContext } from '@/trpc/init';
import { appRouter } from '@/trpc/routers/_app';

const handler = (req: Request) =>
  fetchRequestHandler({
    endpoint: '/api/trpc',
    req,
    router: appRouter,
    createContext: createTRPCContext,
  });

export { handler as GET, handler as POST };
```

**这个文件做了什么事：**

1. 使用 Next.js App Router 的 Route Handler
2. 接收所有 GET 和 POST 请求
3. 调用 `fetchRequestHandler` 处理请求
4. 将请求路由到对应的 Procedure

**endpoint 的作用：**

`endpoint: '/api/trpc'` 定义了 API 的基础路径，实际调用时：

```
GET /api/trpc/voices.getAll
POST /api/trpc/voices.delete
```
**feachRequestHandle 的作用：**

`fetchRequestHandler` 是 tRPC 的 Fetch 适配器，用于处理 HTTP 请求。

``` ts
fetchRequestHandler({
  endpoint: '/api/trpc',  // API 路径前缀
  req,                     // Request 对象
  router: appRouter,       // 你的 tRPC 路由
  createContext: createTRPCContext,  // 创建 Context 的函数
});
```

## 2.7 思考题

1. 为什么要区分 `authProcedure` 和 `orgProcedure`？
  因为有声音创建的时候分组织一起创建和个人创建
2. 如果不使用中间件，直接在每个 Procedure 中检查认证，会有什么问题？
  每次验证都需要写相同的检查逻辑，代码重复；检查逻辑变更时需要修改所有的procedure，不便于维护
3. Sentry 中间件 `attachRpcInput: true` 对调试有什么帮助？
  在错误事件中附带请求参数
